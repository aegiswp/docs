## Overview

Aegis uses an inline assets system to conditionally load CSS and JavaScript only when needed. This is the core mechanism behind the theme's zero-asset-by-default philosophy.

## How It Works

1. **Detection** — Aegis scans page content for blocks at render time
2. **Matching** — Identifies which assets each block needs via registered services
3. **Loading** — Injects only required CSS/JS inline or as enqueued assets

## Architecture

The system consists of 6 files in `vendor/aegis/framework/src/InlineAssets/`:

| File | Purpose |
|------|---------|
| `AssetsTrait.php` | Shared trait for storing inline CSS strings, files, callbacks, and localized data |
| `Inlinable.php` | Interface for services that provide inline CSS |
| `Scriptable.php` | Interface for services that provide localized script data |
| `Styleable.php` | Interface for services that provide inline styles |
| `Scripts.php` | Manages script enqueuing and `wp_localize_script` data |
| `Styles.php` | Manages inline CSS output via `wp_add_inline_style` |

### Interfaces

Services implement one or more interfaces to declare their asset needs:

```php
// Service provides inline CSS
class MyService implements Styleable {
    public function styles(Styles $styles): void {
        $styles->add_string('my-block', '.my-block { color: red; }');
    }
}

// Service provides localized script data
class MyService implements Scriptable {
    public function scripts(Scripts $scripts): void {
        $scripts->add_data('myConfig', ['key' => 'value']);
    }
}
```

## AssetsTrait API

The `AssetsTrait` provides four methods for registering assets:

### `add_string(string $handle, string $css)`

Add a raw CSS string to be output inline:

```php
$styles->add_string('my-block', '.my-block { display: grid; }');
```

### `add_file(string $handle, string $path)`

Add a CSS file to be read and output inline:

```php
$styles->add_file('my-block', __DIR__ . '/css/my-block.css');
```

### `add_callback(string $handle, callable $callback)`

Add a callback that returns CSS. Called lazily at output time:

```php
$styles->add_callback('dynamic-css', function(): string {
    return '.dynamic { color: ' . get_option('my_color') . '; }';
});
```

### `add_data(string $handle, array|Closure $data, array $deps = [], bool $condition = true)`

Add localized data for `wp_localize_script`. Supports closures for deferred evaluation:

```php
// Eager evaluation (data built immediately)
$scripts->add_data('myConfig', ['key' => 'value']);

// Deferred evaluation (closure resolved at enqueue time)
$scripts->add_data('myConfig', fn() => [
    'label' => esc_html__('Translated string', 'my-textdomain'),
], [], is_admin());
```

<Note>
Closure support was added to prevent early translation loading errors. When `add_data()` receives a `Closure`, it is stored as-is and only resolved when `get_data()` is called at enqueue time (after the `init` hook).
</Note>

## Conditional Loading

Assets are only loaded when their associated block is present on the page. The `ServiceProvider` registers services and calls their `styles()` / `scripts()` methods during theme setup, but the actual CSS/JS output is deferred to render time.

## Benefits

- **Performance** — No unused CSS/JS loaded
- **Efficiency** — Smaller page sizes
- **Speed** — Faster page loads
- **No FOUC** — Inline CSS is output in `<head>` before content renders

<Tip>
This zero-asset-by-default approach is key to Aegis's performance philosophy. The theme loads 0 CSS, 0 JS, and 0 fonts by default — everything is conditional.
</Tip>
